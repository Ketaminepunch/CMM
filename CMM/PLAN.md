# "call me maybe" — Function Calling via Constrained Decoding

## Context

This is the 42 School "call me maybe" project (subject at `/sgoinfre/vsack/CMM/en.subject.pdf`). Goal: build a CLI tool that reads natural-language prompts, picks the right function from a provided schema, and extracts correctly-typed arguments — using a small local LLM (Qwen3-0.6B) — while **guaranteeing** 100% valid, schema-compliant JSON via **hand-built constrained decoding** (token-by-token logit masking). Prompting the model and hoping for valid JSON is explicitly forbidden by the subject.

Repo root for the submission: **`/sgoinfre/vsack/CMM/CMM`** — an empty `git init`'d repo (branch `master`, no commits, no remote), solo project, 42 login `vsack`. The provided `llm_sdk.zip` (in the parent folder) has already been inspected — its `Small_LLM_Model` class wraps `transformers`/`torch`/`huggingface_hub` and exposes only: `encode(text) -> Tensor`, `decode(ids) -> str`, `get_logits_from_input_ids(ids) -> list[float]`, `get_path_to_vocab_file() -> str`, plus `get_path_to_merges_file()` / `get_path_to_tokenizer_file()` (public, undocumented in the subject, useful for the bonus). Live verification confirmed `Qwen/Qwen3-0.6B`'s `vocab.json` is a standard GPT‑2-style byte-level BPE vocab (`{token_str: id}`, 151,643 entries, `Ġ` = leading space) — this is what makes a from-scratch token↔string map (no private SDK internals) feasible.

## Core design: the constrained decoder

The model must only ever produce `{"name": "<fn>", "parameters": {...}}` (compact JSON, no whitespace). `prompt` is known already and gets injected when the final output file is assembled — it's never generated by the model, which shrinks the grammar.

**Grammar = character-level pushdown automaton (`src/grammar.py`)**, not a general JSON parser:
- Fixed literal spans (`{"name":"`, `,"parameters":{`, per-key `"paramX":`, closing `}}`) — single valid path.
- `ENUM` state (function name, and `true`/`false` for booleans) — a trie over the allowed literal strings; only trie-consistent chars are valid; closing quote only allowed at a trie node marking a complete word. Once the name closes, the automaton **commits** to that function and swaps in its specific parameter schema for the rest of generation — this is how "the LLM chooses the function" without heuristics: the model's masked logits drive the trie walk.
- `NUMBER` state — JSON number grammar (`-?digits(.digits)?`, exponents intentionally out of scope, documented as a limitation).
- `STRING` state — any char except unescaped `"`/control chars, with a proper escape sub-state (`\"`, `\\`, `\n`, `\uXXXX`, …).
- The automaton has an explicit **ACCEPT** state reached only when structurally complete — decoding stops there deterministically. We never rely on the model's EOS token, so 100% valid JSON is structural, not probabilistic.

**Two key performance optimizations (core, not bonus — needed to hit the "<5 min" requirement with no KV-cache in `get_logits_from_input_ids`, which reprocesses the full sequence every call):**
1. **Skip the model entirely for forced literal spans.** When the grammar state has exactly one valid continuation (fixed punctuation/keys), tokenize that literal fragment via `sdk.encode()` and append its ids directly — no forward pass. The model (and our mask/argmax logic) is only invoked at genuine decision points: function-name enum, argument values, delimiter/continuation choices.
2. **Memoize valid-token-id sets per grammar-state signature.** Many states recur identically across prompts (e.g. `,"parameters":{` chain, or "expect one of these 3 function names") — compute the mask once, cache by state signature, reuse. Combined with `numpy` for the masking/argmax step.

**Tokenizer layer (`src/tokenizer_vocab.py`)**: loads `vocab.json` via `sdk.get_path_to_vocab_file()`, builds `id -> str` (GPT‑2 byte-decoder table, for per-token grammar validity checks) and a `decode_ids(ids) -> str` that buffers raw bytes across the whole accepted-token sequence and UTF‑8-decodes once at the end (correct at multi-byte boundaries, and means we never call `sdk.decode()` in the main path — partially satisfies the "avoid decode/encode, use vocab file" bonus point for free). `sdk.encode()` is still used for the prompt and forced-literal fragments (pragmatic; a from-scratch BPE encoder using `get_path_to_merges_file()` is flagged as a follow-up bonus, not built now).

## Build order

1. **Scaffold**: unzip `llm_sdk.zip` into repo root, write `pyproject.toml`/Makefile/`.gitignore`, empty `src/` module stubs, commit the subject's example JSON into `data/input/`. Verify `uv sync` and `make run` execute (even as a no-op).
2. **Models + IO**: pydantic schemas, safe JSON load/save with clear error messages for missing/malformed input files (subject explicitly warns these may be invalid or missing).
3. **Tokenizer layer**: `tokenizer_vocab.py` against the real downloaded `vocab.json`; sanity-check byte-decode round-trips.
4. **Grammar**: the PDA — literals, enum trie (function names + booleans), number/string sub-grammars, state-signature memoization. Unit-test with fake logits (no model needed) before wiring the real SDK.
5. **Decoder loop**: wire `Small_LLM_Model` + grammar + vocab; forced-literal fast path; masked-argmax decision path; hard max-token safety cap with a clear internal error (caught upstream, never crashes the run).
6. **Prompt builder + pipeline + CLI**: instantiate the model once and reuse it across all prompts (major perf win); per-prompt try/except so one bad prompt can't kill the whole run; assemble and write `function_calling_results.json`.
7. **README.md**: italic first line (`*This project has been created as part of the 42 curriculum by vsack.*`), Description, Instructions, Resources (+ AI-usage disclosure), plus the project-specific required sections: Algorithm explanation, Design decisions, Performance analysis, Challenges faced, Testing strategy, Example usage.
8. **End-to-end verification**: `make install && make lint && make run` against the real Qwen3-0.6B (network confirmed working, ~1.2GB download) on the subject's example data; check output validity/schema/timing (<5 min). Fix issues found.
9. **Bonus** (only after mandatory is verified working, time permitting): flagged in README as follow-ups — from-scratch BPE encoder to drop `sdk.encode()` too, caching/batching, nested-argument support, a small visualization of the generation process.
